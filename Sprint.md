## 제품 목표 요약

지원바다는 정부지원금 정보를 한곳에 모아 쉽고 빠르게 찾게 해주는 가이드형 웹사이트다.

초기 제품의 핵심 가치는 아래 4가지다.
- 쉬움: 어려운 정책 용어를 사용자 친화적 문장으로 재구성
- 빠름: 검색, 카테고리 탐색, 자격 체크로 빠른 탐색 제공
- 최신성: 크롤링과 자동 배포로 데이터 상시 업데이트
- 신뢰성: 공식 출처 링크와 업데이트 날짜를 명시

---

## 구현 원칙

### 1. 기술 원칙

- 프레임워크: Next.js 15 App Router
- 배포: OpenNext + Cloudflare Pages/Workers
- 스타일: Tailwind CSS + 디자인 토큰
- 데이터: `data/grants.json` 기반
- 크롤링: Python + BeautifulSoup4
- 자동화: GitHub Actions

### 2. UX/UI 원칙

- 홈과 주요 랜딩은 Apple식 풀블리드 섹션 리듬을 따른다.
- 인터랙션 강조 색상은 `#0066cc` 단일 컬러를 사용한다.
- 배경은 `white / parchment / dark` 3축으로 구성한다.
- 카드형 정보는 필요하지만, 일반 SaaS처럼 강한 카드 UI로 가지 않는다.
- 카드 경계는 주로 `hairline border`로 표현하고, 그림자는 있더라도 매우 약하게 제한한다.
- 본문 가독성과 검색 편의성을 최우선으로 둔다.

### 3. 정보 설계 원칙

- 모든 지원금은 단일 데이터 스키마로 관리한다.
- 상세 페이지는 동일한 정보 구조를 공유한다.
- 카테고리, 토픽, 검색, 자격 체크는 모두 같은 원천 데이터에서 파생된다.
- SEO용 페이지와 사용자 탐색용 페이지를 분리하지 않고 하나의 구조에서 해결한다.

## 전체 일정 개요

초기 구현은 6개 스프린트로 나눈다.

1. Sprint 1: 프로젝트 기반 구축
2. Sprint 2: 공통 디자인 시스템과 레이아웃
3. Sprint 3: 핵심 콘텐츠 구조와 홈/목록/상세 페이지

5. Sprint 5: 데이터 수집 자동화와 운영 파이프라인
6. Sprint 6: SEO, 정책 페이지, 수익화 준비, 안정화

#### 2-1. 디자인 토큰 정리

- 폰트 스케일 반영

#### 2-2. 공통 레이아웃
- 루트 레이아웃 구성

- 디자인 토큰이 코드로 정리되어 재사용 가능하다.

## Sprint 5. 데이터 수집 자동화와 운영 파이프라인

### 목표

콘텐츠를 수동 입력에만 의존하지 않도록 데이터 수집과 배포 흐름을 자동화한다.

### 기능 범위

- 크롤러 1차 구현
- 데이터 정규화
- GitHub Actions 자동 실행
- 변경 시 재배포 트리거

### 세부 작업

#### 5-1. 크롤러 구조 설계
- 소스별 수집 모듈 분리
- 공통 fetch/parser/normalizer 구조 설계
- 오류 로그 포맷 정의

#### 5-2. 1차 수집 대상 구현
- 복지로
- 고용24
- 이후 확장 가능하도록 인터페이스 통일

#### 5-3. 데이터 정규화
- 외부 데이터를 grants 스키마에 맞게 변환
- 중복 감지 규칙 정의
- 상태값 산정 로직 정의
- `last_updated`, `source_url` 보존

#### 5-4. GitHub Actions
- 매일 새벽 실행 cron 설정
- Python 환경 설치
- 크롤러 실행
- `data/grants.json` 변경 시 커밋/푸시
- Cloudflare Deploy Hook 호출

#### 5-5. 운영 안정성
- 크롤링 실패 시 graceful fallback
- 일부 소스 실패 시 전체 중단 방지
- robots.txt 및 요청 간격 준수

### 완료 기준

- 최소 1~2개 소스에서 데이터를 수집해 `grants.json`을 생성할 수 있다.
- 자동화 워크플로가 동작한다.
- 데이터 변경 시 사이트가 재배포된다.

## 위험 요소와 대응

### 1. 데이터 품질 문제
- 위험: 소스 사이트 구조 변경으로 파싱 실패

- 위험: README 예시를 그대로 따라가면 Apple 스타일보다 무거워질 수 있음
- 대응: 카드보다 섹션 리듬 우선, border 중심 표현, 그림자 최소화

### 3. 필터 조건과 실제 데이터 불일치
- 위험: 자격 체크 설문과 grants 데이터가 정확히 매핑되지 않을 수 있음
- 대응: 초기에는 단순 매핑으로 시작하고, 데이터 스키마를 점진 확장

### 4. SEO 페이지 수 부족
- 위험: 초기 애드센스/색인 효율 저하
- 대응: 우선순위 높은 20개 지원금 상세 페이지를 먼저 확보

### 5. Cloudflare/OpenNext 설정 이슈
- 위험: 로컬은 되지만 배포 환경에서 실패 가능
- 대응: Sprint 1 후 바로 첫 배포 리허설 수행
